## 테니스 동호회 행사 앱 개발 계획서

### 1. 프로젝트 개요
- **프로젝트 목적**: 테니스 동호회 행사(총 12명 참가)의 경기 진행 상황과 점수를 실시간으로 공유하고, 마니또 비밀 투표까지 지원하는 웹 애플리케이션을 Node.js 기반으로 개발한다.
- **대상 사용자**: 행사에 참여하는 12명 전원 (관리자 개념 없이 누구나 입력/조회 가능).
- **핵심 가치**
  - 종이 대진표 대신 스마트폰/태블릿으로 실시간 공유
  - 경기 점수 및 합계 자동 계산
  - 마니또 비밀 투표로 행사 재미 강화

### 2. 요구사항 정리

#### 2.1 공통
- **참가자 수**: 12명 고정.
- **대진표 구조**: 제공된 이미지의 시간/코트/팀 구성과 동일하게 고정 편성.
- **권한**: 로그인만 하면 누구나 점수 입력 및 합산 가능(간단한 행사용). 이름을 입력해 사이트에 입장할 수 있으며, **동시 접속 인원은 최대 35명**으로 제한한다.
- **디자인 컨셉**
  - 최신 웹 디자인 트렌드(네오 브루탈리즘 또는 글래스모피즘 기반 카드 UI)
  - 반응형 레이아웃(모바일 우선, 태블릿/노트북 지원)
  - 행사용 커버 이미지를 로그인 화면과 대시보드 상단 히어로 섹션에 공통 사용.

#### 2.2 기능 상세
- **기능 0 – 대시보드**
  - 이미지의 대진표 레이아웃을 참고하여, 시간대별(R1~R6), 코트별(1코트/2코트), 팀/플레이어 배치를 표 형태로 시각화.
  - 각 셀에는 팀 구성(2인)과 점수 입력 필드를 표시.
  - 현재까지 입력된 점수와 승패/무승부 여부를 색상으로 구분(예: 승리=초록, 패배=빨강, 미진행=회색).

- **기능 1 – 로그인(이름 기반)**
  - 별도 회원가입 없이, **이름 입력만으로 로그인**.
  - 12명 참가자 이름 목록을 사전에 설정해두고, 드롭다운 선택 + 간단한 PIN(선택)으로 인증할 수 있게 고려.
  - 로그인 후 상단 바에 현재 사용자 이름 표시, 같은 기기에서는 세션/쿠키를 이용해 자동 로그인 유지(행사 동안).

- **기능 2 – 점수 입력**
  - 각 경기 셀에 두 팀의 점수 입력 필드(숫자 입력).
  - 점수 변경 시 자동 저장(또는 `저장` 버튼) 및 실시간 반영.
  - 특정 경기의 점수 변경 내역은 마지막 수정자 이름과 시간 로그를 소형 텍스트로 표기(선택 사항).

- **기능 3 – 합산**
  - `합산하기` 버튼 클릭 시,
    - 각 참가자별 총 경기 수, 승/패/무, 총 득점/실점, 득실차 계산.
    - 참가자별 랭킹 테이블로 표시.
  - 합산 결과는 별도 섹션(예: `오늘의 성적`)에 카드 형태/테이블로 표시.

- **기능 4 – 마니또 비밀 투표**
  - 로그인한 참가자는 1인 1표로 **오늘의 마니또**를 비밀 투표.
  - 자기 자신에게는 투표 불가.
  - 투표 내용은 관리자만 조회 가능하다고 가정하고, UI 상에서는 **집계 결과만** 보여주거나, 행사 종료 시에만 공개 버튼을 통해 공개.
  - 투표 완료한 사용자는 다시 수정할 수 없도록 제한(선택: 수정 허용 옵션 플래그).

- **기능 5 – 커버 이미지 적용**
  - 제공된 대진표 이미지를 참고해서, 별도의 **브랜드 커버 이미지**(행사명, 날짜 포함)를 생성/적용한다고 가정.
  - 로그인 화면: 풀스크린 히어로 배경 + 반투명 카드에 로그인 폼 배치.
  - 대시보드: 상단 히어로 섹션(행사명, 일정, 장소, 마니또 안내 텍스트) 배경으로 커버 이미지 사용.

### 3. 기술 스택 및 아키텍처

- **백엔드**
  - Node.js (LTS)
  - Express.js 기반 REST API
  - 세션 관리: express-session 또는 JWT(간단 행사이므로 서버 세션 우선)

- **프론트엔드**
  - React + Vite (또는 Next.js) – SPA 형태로 개발
  - 상태관리: React Query 또는 간단한 Context/Reducer
  - 스타일링: Tailwind CSS + 커스텀 컴포넌트(최신 디자인 시스템 느낌 구현)

- **데이터 저장**
  - **서버 재시작·장애 상황에서도 데이터 유지**를 위해 인메모리 저장은 사용하지 않는다.
  - 기본 저장소로 **SQLite 파일 DB + Prisma ORM** 조합을 사용하여, 경기/점수/투표 데이터가 디스크에 안전하게 기록되도록 한다.
  - 운영 환경에서는 SQLite DB 파일의 **정기 백업(예: 일정 간격으로 다른 경로/클라우드에 복사)** 전략을 함께 사용해, 서버/노트북이 꺼져도 데이터를 복구할 수 있도록 고려한다.
  - 향후 여러 행사, 동시 접속이 많아질 경우 PostgreSQL 같은 외부 RDBMS로 마이그레이션 가능하도록 스키마를 설계한다.

- **배포**
  - 단일 Node.js 서버로 프론트 정적 파일 서빙 + API 제공.
  - 추후 필요 시 프론트/백엔드 분리 및 클라우드(예: Render, Railway 등) 배포 고려.

### 4. 도메인 모델 설계

- **User (Participant)**
  - id
  - name
  - optionalPin

- **Match**
  - id
  - round (R1~R6)
  - court (1 or 2)
  - startTime, endTime
  - teamA: 참가자 id 배열(2명)
  - teamB: 참가자 id 배열(2명)
  - scoreA, scoreB
  - lastUpdatedBy (User id)
  - lastUpdatedAt

- **Vote**
  - id
  - voterId (User id)
  - targetId (User id)
  - createdAt

### 5. 주요 화면 설계

- **로그인 화면**
  - 커버 이미지를 배경으로 사용하는 풀스크린 레이아웃.
  - 중앙 카드: 반투명 유리(glassmorphism) 스타일, 소프트 섀도우와 라운드 코너.
  - 요소
    - 행사 타이틀, 날짜, 간단 설명
    - 이름 선택 드롭다운 또는 이름 입력 필드
    - (선택) 간단 PIN 입력
    - `입장하기` 버튼

- **대시보드 화면**
  - 상단: 커버 이미지를 배경으로 한 히어로 영역(행사명, 현재 라운드 안내, 마니또 설명).
  - 중단: 라운드/코트별 대진표 그리드
    - 열: 시간/라운드, 1코트, 2코트
    - 셀: 팀 구성 + 점수 입력 필드 + 진행 상태 배지
  - 하단:
    - `합산 결과` 카드: 참가자별 순위/득실/승률
    - `마니또 투표` 섹션: 대상자 선택 드롭다운 + 투표 버튼 (이미 투표했다면 결과 요약만 표시).

### 6. API 설계 (초안)

- **인증 / 세션**
  - `POST /api/login` – 이름(및 PIN)을 받아 세션 생성.
  - `POST /api/logout` – 세션 삭제.
  - `GET /api/me` – 현재 로그인된 사용자 정보 조회.

- **대진표 및 경기**
  - `GET /api/matches` – 전체 대진표(경기 리스트) 조회.
  - `PUT /api/matches/:id/score` – 특정 경기 점수 업데이트(세션 사용자 정보를 lastUpdatedBy로 기록).

- **합산 결과**
  - `GET /api/summary` – 참가자별 승/패/무, 득점/실점, 랭킹 반환.

- **마니또 투표**
  - `POST /api/votes` – { targetId }로 투표 생성(로그인 사용자 기준).
  - `GET /api/votes/summary` – 집계 결과(득표수 상위 N명) 조회.

### 7. 개발 순서

1. **프로젝트 초기 설정**
   - Node.js + Express 기본 서버 구성
   - Vite + React + Tailwind 초기 세팅
   - 공통 타입 정의 (User, Match, Vote)
2. **대진표/데이터 모델 구현**
   - SQLite + Prisma 초기 세팅 및 마이그레이션( User, Match, Vote 테이블 ) 적용
   - 12명 참가자 및 경기 편성을 **DB 시드 데이터**로 정의하여, 서버를 재시작해도 동일한 데이터가 유지되도록 한다.
   - `/api/matches` 엔드포인트 구현
3. **로그인 플로우**
   - 간단 세션 기반 로그인 API + 프론트 로그인 화면
4. **대시보드 UI + 점수 입력**
   - 이미지 구조를 반영한 그리드 UI 구현
   - 점수 입력 및 저장/실시간 반영
5. **합산 로직 및 결과 화면**
   - 백엔드에서 합산/랭킹 계산 API 구현
   - 프론트에서 테이블/카드형 UI로 표시
6. **마니또 비밀 투표**
   - 투표 API 및 제한 로직(1인 1표) 구현
   - 투표 UI 및 집계 결과 UI 구현
7. **디자인 다듬기 및 배포**
   - 커버 이미지 적용, 컬러/타이포 조정
   - 간단한 배포(또는 로컬용 실행 스크립트 작성)

### 8. 안정성 및 로깅/에러 처리

- 공통 에러 처리 미들웨어를 구성하여 API 에러 응답 포맷을 통일한다.(status code, message, details 형식)
- 404/500 등 기본 에러 페이지 및 JSON 응답을 정의한다.
- 점수 업데이트, 투표 생성/실패 등 핵심 액션에 대해 최소한의 서버 로그를 남겨, 행사 중 문제 발생 시 원인을 추적할 수 있도록 한다.

### 9. 테스트 및 리허설

- 합산 로직, 1인 1표 제한, 로그인/세션 유지 등 핵심 도메인에 대해 간단한 유닛 테스트 또는 수동 테스트 시나리오를 작성한다.
- 실제 행사 환경(모바일 브라우저, 여러 사용자가 동시에 접속)에서 대시보드·점수 입력·투표까지 한 번 전체 플로우를 리허설한다.

### 10. 배포 및 운영 체크리스트

- `npm run dev`, `npm run build`, `npm run start` 등 개발/운영용 스크립트를 정리하고 README에 실행 방법을 명시한다.
- SQLite DB 파일 위치와 간단한 백업 방법(수동/스크립트)을 정리한다.
- 서버 포트, 세션 비밀키, DB 경로 등 환경 변수 목록을 정리해 두고, 행사 당일 재시작 시 따라야 할 간단한 운영 절차를 문서화한다.
